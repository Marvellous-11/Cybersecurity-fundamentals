# 🛡️ SOC Analyst Investigation (Network Alert Analysis)

## Project Overview
This project simulates the role of a Security Operations Center (SOC) analyst investigating network alerts generated by an Intrusion Detection System (IDS) using Suricata logs.

The goal is to detect, analyze, and interpret suspicious network activity using real log data.

---

## Scenario
A network monitoring system generated alerts indicating HTTP traffic activity. As a SOC analyst, I investigated the alerts to determine whether the activity was malicious or normal.

---

## Tools Used
- Suricata (IDS)
- tcpdump
- jq (log parsing)
- Linux command line

---

## Detection Rule
A custom Suricata rule was used to detect HTTP GET requests:

```bash
alert http $HOME_NET any -> $EXTERNAL_NET any (msg:"GET on wire"; flow:established,to_server; content:"GET"; http_method; sid:12345; rev:3;)
```

---

## Data Collection

### Running Suricata
```bash
sudo suricata -r sample.pcap -S custom.rules -k none
```

### Generated Logs
- eve.json (detailed structured logs)
- fast.log (alert summary)
- stats.log
- suricata.log

---

## Alert Evidence

### fast.log
```bash
[1:12345:3] GET on wire {TCP} 172.21.224.2 -> 142.250.1.139:80
```

This shows:
- Source IP: 172.21.224.2
- Destination IP: External server
- Protocol: TCP
- Activity: HTTP GET request

---

## Log Analysis

### Extracting Key Fields
```bash
jq -c "[.timestamp,.flow_id,.alert.signature,.proto,.dest_ip]" eve.json
```

### Sample Output
```bash
["2022-11-23T12:38:34","GET on wire","TCP","142.250.1.139"]
```

---

## Findings

- Multiple HTTP GET requests were detected
- Traffic was directed to an external web server
- Response included HTTP 301 redirect (to HTTPS)
- No malicious payload detected

---

## Analysis

### Is this malicious?
No — this activity appears to be normal web traffic.

### Why?
- HTTP GET requests are standard for web browsing
- Destination is a legitimate domain
- Redirect to HTTPS is expected behavior
